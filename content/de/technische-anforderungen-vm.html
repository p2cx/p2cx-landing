Technische Anforderungen
VM für p2cx Apps

Greif-Velox möchte eigene Web-Applikationen auf einer VM im eigenen Netzwerk betreiben. Dieses Dokument beschreibt, was die VM braucht, welche Zugänge p2cx benötigt und wie wir gemeinsam den Betrieb sicherstellen.

Dokument für: IT-Dienstleister Greif-Velox · Stand: 2026-05-01 · p2cx GmbH
Überblick

Zusammenfassung

Die Apps basieren auf einem modernen Open-Source-Stack (Node.js, PostgreSQL). Wir empfehlen Docker als Deployment-Methode — das macht Updates einfach, Rollbacks sicher und den Betrieb reproduzierbar.

Moderner Stack

Node.js, PostgreSQL, Docker — Open Source, bewährt, leichtgewichtig.

Docker Deployment

Einfache Updates, sichere Rollbacks, reproduzierbarer Betrieb.

Rein intern

Kein eingehender Internet-Zugang nötig — nur im Firmennetzwerk erreichbar.

Abschnitt 4

ERP-Anbindung (Penta)

Die OPOS-App braucht Zugriff auf Daten aus dem ERP-System. Die VM muss im internen Netzwerk den Penta-Datenbankserver erreichen können.

Empfohlener Ansatz: Read-Only SQL-Zugriff

AspektDetails
Datenbank-TypVermutlich Microsoft SQL Server (bei Penta üblich)
Port1433 (MSSQL Standard)
RichtungVM → Penta-DB-Server (intern, kein Internet)
BerechtigungenRead-Only — die App liest nur, schreibt nie ins ERP
DB-UserDedizierter User mit minimalen Rechten (nur SELECT auf benötigte Tabellen/Views)

Varianten

VarianteBeschreibungVorteilNachteil
Direkte SQL-AbfrageVM verbindet sich direkt auf Penta-DBEchtzeit-Daten, einfachDB-Last durch Abfragen
Staging-View/KopieIT-DL stellt eine replizierte Kopie/View bereitKeine Last auf Produktiv-DBDaten nicht ganz live
Datei-Export (CSV)Penta exportiert regelmäßig auf NetzlaufwerkKeine DB-Verbindung nötigManuell, nicht live, fehleranfällig

Empfehlung: Direkte SQL-Abfrage (Read-Only) oder Staging-View. Datei-Export nur als Notlösung.

Sicherheit

  • Die App liest nur — kein Schreibzugriff auf Penta, niemals
  • Der DB-User hat nur SELECT-Rechte auf definierte Tabellen
  • Verbindung läuft nur intern (kein Internet-Zugang zur Penta-DB)
  • Credentials liegen verschlüsselt in der Docker-Konfiguration (nicht im Code)

Was der IT-Dienstleister bereitstellen muss

  • IP/Hostname des Penta-Datenbankservers
  • Port (vermutlich 1433)
  • Read-Only Datenbank-User anlegen (nur SELECT-Rechte)
  • Firewall: VM darf intern auf den DB-Server zugreifen
  • Klären: Welche Tabellen/Views werden benötigt? (stimmen wir gemeinsam ab)

Abschnitt 5

Remote-Zugang für p2cx

p2cx braucht SSH-Zugang zur VM für Ersteinrichtung und Konfiguration, Deployments und Updates, Fehleranalyse bei Problemen sowie Monitoring-Einrichtung.

Empfehlung: Reverse-SSH-Tunnel

Kein Drittanbieter, kein VPN-Client, keine US-Dienste. Die VM baut selbst eine ausgehende Verbindung zum p2cx-Server auf.

Reverse-SSH-TunnelEmpfohlen
Greif-Velox VM → ausgehende SSH-Verbindung → p2cx-Server (EU, IONOS) ← p2cx greift zu
  • Kein eingehender Port muss auf der Firewall geöffnet werden
  • Kein VPN-Client, keine Drittanbieter-Software
  • Verbindung ist SSH-verschlüsselt (AES-256)
  • IT-DL kann die Verbindung jederzeit kappen
  • Ermöglicht automatisiertes Deployment per GitHub Actions
Warum das sicher ist:
  • Kein offener Port auf der VM (kein Angriffspunkt)
  • p2cx erreicht nur die VM, nicht das restliche LAN
  • Keine Metadaten bei US-Firmen
  • Key-basierte Authentifizierung (keine Passwörter)
  • Eingeschränkter User (nur Docker + Logs)
Wer hat welchen Key?
KeyErstellt vonLiegt aufZweck
Tunnel-Key (private)IT-DLGreif-Velox VMVM authentifiziert sich beim p2cx-Server
Tunnel-Key (public)IT-DLp2cx-Serverp2cx-Server erkennt die VM
p2cx-Key (private)p2cxp2cx-Rechnerp2cx authentifiziert sich auf der VM
p2cx-Key (public)p2cxGreif-Velox VMVM erkennt p2cx

Kein Private Key verlässt jemals den Rechner auf dem er erstellt wurde.

GitHub Actions: Automatische Deployments
Code Push → Tests → Docker Image bauen → SSH über Tunnel → docker pull + restart → Health-Check

Die VM braucht nur ausgehenden Zugang zu:

  • Port 22 (SSH zum p2cx-Server, für den Tunnel)
  • Port 443 (HTTPS zu ghcr.io, für Docker-Image-Pull)
systemd Service auf der VM # /etc/systemd/system/p2cx-tunnel.service [Unit] Description=SSH Tunnel zu p2cx After=network-online.target [Service] ExecStart=/usr/bin/ssh -N -R 2222:localhost:22 \ -i /etc/ssh/tunnel_key \ -o ServerAliveInterval=30 \ -o ServerAliveCountMax=3 \ tunnel@p2cx-server-ip Restart=always RestartSec=10 [Install] WantedBy=multi-user.target

Alternative Optionen

Tailscale (einfach, aber US-Anbieter)
p2cx ← Tailscale-Netzwerk (P2P) → VM
  • Peer-to-Peer VPN, Daten gehen direkt (nicht über USA)
  • Aber: Koordinations-Server in USA/Kanada (Metadaten)
  • 5 Minuten Setup, kostenlos bis 3 Geräte

Vorteil: Schnellstes Setup

Nachteil: US-Metadaten, Drittanbieter

VPN-Zugang (klassisch)
p2cx → VPN → Firmennetzwerk → SSH auf VM
  • IT-Dienstleister richtet VPN-Account für p2cx ein
  • Standard bei den meisten Firmen

Vorteil: IT-DL hat volle Kontrolle

Nachteil: Nicht automatisierbar, kein CI/CD. Oft proprietärer Client + 2FA nötig.

SSH-Einrichtung auf der VM

Shell # Benutzer für p2cx anlegen sudo adduser p2cx --disabled-password sudo usermod -aG docker p2cx # SSH-Key hinterlegen (kein Passwort-Login!) sudo mkdir -p /home/p2cx/.ssh sudo nano /home/p2cx/.ssh/authorized_keys # → p2cx liefert den Public Key # Passwort-Login deaktivieren (Sicherheit!) sudo sed -i 's/#PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config sudo systemctl restart sshd

Was p2cx mit dem Zugang macht (und was nicht)

ErlaubtNicht erlaubt
Docker-Container starten/stoppen/aktualisierenNetzwerk-Konfiguration ändern
Logs lesenAndere Server/Dienste im Netzwerk zugreifen
Datenbank-Operationen (Migrationen, Backups)Firewall-Regeln ändern
Monitoring einrichtenNutzer-Accounts anlegen/löschen
Fehleranalyse und DebuggingBetriebssystem-Updates (macht IT-DL)

Wie das Monitoring funktioniert

Apps auf VM → /api/health → p2cx Monitoring (myp2cx) | +-- Dashboard: Grün/Rot pro App +-- Alerting: E-Mail bei Ausfall +-- Historie: Uptime-Verlauf
Option A: p2cx fragt die VM direkt ab

Braucht VPN/Tailscale. Einfachste Lösung: p2cx ruft /api/health jeder App auf. Ergebnisse landen im myp2cx-Dashboard.

Option B: VM meldet sich bei p2cx

Kein eingehender Zugang nötig. Kleiner Agent auf der VM schickt Health-Daten an p2cx. Push statt Pull — funktioniert ohne VPN. VM braucht nur ausgehenden HTTPS-Zugang.

Wartung und Updates

OS-Updates durch den IT-Dienstleister, App-Updates durch p2cx — mit Qualitätssicherung.

OS-Updates (IT-Dienstleister)

Shell # Regelmäßig (monatlich oder häufiger): sudo apt update && sudo apt upgrade -y # Docker aktualisieren: sudo apt install --only-upgrade docker-ce docker-ce-cli

App-Updates (p2cx oder automatisch)

Shell # Manuell (p2cx per SSH): cd /opt/greifvelox docker compose pull docker compose up -d # Oder automatisch per GitHub Actions: # Push auf main → neues Image wird gebaut → Webhook an VM → auto-update

Update-Ablauf mit Qualitätssicherung

Code-Änderung → Tests (automatisch) → Docker Image bauen → Image auf VM ziehen → Health-Check → Fertig → Health-Check FEHLGESCHLAGEN → Rollback auf vorheriges Image
Abschnitt 8

Checkliste für den IT-Dienstleister

Schritt für Schritt — was vor und nach der Einrichtung zu tun ist.

Vor dem ersten Termin mit p2cx
  • VM bereitstellen (Ubuntu 24.04 LTS, 4 vCPU, 8 GB RAM, 80 GB SSD)
  • SSH-Zugang einrichten (Key-basiert, Benutzer p2cx)
  • Docker + Docker Compose installieren
  • Ausgehenden HTTPS-Zugang (Port 443) erlauben
  • Ausgehenden SSH-Zugang (Port 22) zum p2cx-Server erlauben
  • SSH-Tunnel-Key generieren und Public Key an p2cx übergeben
  • Tunnel-Service einrichten (systemd, siehe Abschnitt 5)
  • Internes DNS einrichten (optional, aber empfohlen)
  • Backup-Strategie festlegen (Ziel, Frequenz, Aufbewahrung)
  • Ansprechpartner benennen (wer ist erreichbar bei Fragen?)
Nach der Einrichtung durch p2cx
  • Docker-Container laufen (docker ps zeigt alle Apps)
  • Apps erreichbar im Browser (intern)
  • Health-Checks grün (curl http://localhost:3001/api/health)
  • Backup-Cronjob eingerichtet und getestet
  • Monitoring-Zugang für p2cx funktioniert
  • SSL-Zertifikate installiert (HTTPS)
  • Dokumentation übergeben (Ports, Zugänge, Passwörter)
